Agents

FP
Feature Planner opus
Plans cross-repo features with two plan-mode approval cycles. Creates STD-001 spec artifacts.
▾
Skills
feature-planner sdd codebase-researcher
What It Does
  • Gathers requirements through interactive Q&A
  • Detects new feature vs modification of existing behavior
  • Delegates codebase research to both API and Frontend
  • Creates proposal.md, spec-delta.md, and tasks.md
  • Two approval gates: setup plan + feature plan
  • Auto-transitions SDD state to implementation
BE
Backend Engineer opus
Implements API endpoints, business logic, database operations, and tests. Thin routers, fat services.
▾
Skills
backend-endpoint-builder backend-database-ops
Golden Rule
  • Routers are thin: validate input, call service, return response
  • Services are fat: all business logic lives here
  • Models use db.flush() only, never commit
  • LLM calls are OUTSIDE database session blocks
  • Marks tasks in tasks.md as completed
FE
Frontend Engineer opus
Implements React components, pages, hooks, and API integration. Components render, hooks think, services fetch.
▾
Skills
frontend-component-builder frontend-api-integration frontend-state-management
Architecture
  • Components render UI only, no business logic in JSX
  • Hooks manage state and side effects
  • Services handle API calls via apiHandler()
  • Server data in React Query, client state in Redux
  • No `any` types — TypeScript interfaces required
  • Marks tasks in tasks.md as completed

Browser Verification Engineer sonnet
Tests features in Chrome against spec scenarios. Captures screenshots, routes failures to engineers.

Skills
browser-verification
Process

How to Create a Feature
Recommended: Tell Claude what you want to build. The Feature Planner agent activates automatically and walks you through requirements, research, and artifact creation with two approval checkpoints.
Interactive (Feature Planner)
  • 1. Describe your feature to Claude
  • 2. Cycle 1: Agent creates SDD state + change directory, enters plan mode for proposal gate
  • 3. You approve the setup plan
  • 4. Cycle 2: Agent asks questions, researches both codebases, presents findings in plan mode
  • 5. You approve the feature plan
  • 6. Agent writes proposal.md, spec-delta.md, tasks.md and transitions to implementation
  • 7. Handoff: agent tells you which engineer agents to use next
Manual
cp -r changes/.templates changes/{slug}
Copy templates to start a new change
/sdd start {slug}
Or use the SDD command to set up the change directory and state
Spec Artifacts

Every change produces four files in changes/{slug}/

config.yaml
Metadata: name, author, status, target spec, repos affected
proposal.md
What and why. Problem, solution, scope, acceptance criteria.
spec-delta.md
Behavior changes in Given/When/Then. Tagged [API], [Frontend], or [Both].
tasks.md
Implementation checklist. Grouped by API Tasks and Frontend Tasks.
SDD Lifecycle Commands
/sdd start {slug}
Creates changes/{slug}/ with templates. Sets SDD state to proposal.
/sdd status
Shows current cycle, artifact status, task progress (done vs pending).
/sdd implement
Validates artifacts, transitions to implementation. Shows handoff.
/sdd complete {slug}
Moves to archive/, updates history, commits. Run after PR merges.

When Do You Need a Spec?
SituationSpec?Why
New featureYesBehavior needs to be defined
Changing existing behaviorYesDocument what's different
Breaking API changesYesFrontend needs to know
Database schema changesYesStructural changes need tracking
Security-related changesYesHigh-risk needs review
Bug fix (restoring behavior)NoSpec already describes correct behavior
Refactoring (no behavior change)NoUsers don't see anything different
Dependency/config updatesNoNo behavior change
Rule of thumb: If a user would notice the change, write a spec. If they wouldn't, skip it.
Commit Prefixes

Always commit spec before code. The [spec] commit goes first, then [impl].

PrefixWhen
[spec]Spec artifacts committed (proposal, spec-delta, tasks)
[impl]Feature implementation code
[fix]Bug fix restoring specified behavior
[refactor]Code restructuring with no behavior change
[archive]Moving completed change to archive/
[chore]Dependencies, config, docs, CI/CD
[feat]New tooling, agents, skills, or infrastructure
